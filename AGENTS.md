# 에이전트 작업 지침

## 코딩 지침
여러분이 한 번에 작성한 코드는 앞으로 여러 에이전트가 수십, 수백, 수천 번 읽게 될 코드입니다.
개발 작업 시 유지보수성을 위해 아래 원칙을 준수합니다.

- **KISS**: 일관되고 단순한 코드는 요구사항이 늘어나더라도 동작을 예측하기 쉽게 만듭니다.
  디렉터리, 파일, 함수, 클래스, 인자 이름과 반환 타입만 봐도 구현 의도를 어느 정도 예측할 수 있어야 합니다.
- **YAGNI**: 공개 인터페이스를 지키면서 확장하는 것은 언제든지 가능합니다.
  프로젝트 방향이 바뀔 수 있으므로, 불필요한 추상화와 기능은 개발 부채가 됩니다.
- **DRY**: 모든 파일을 읽고 중복 여부를 파악하는 데에는 많은 비용이 듭니다.
  디렉터리, 파일, 함수, 클래스, 메서드 이름은 `rg`로 쉽게 검색할 수 있도록 명확하게 작성합니다.
- **SOLID**: 테스트 가능하고 변경에 강한 구조를 만들기 위한 원칙으로 활용합니다.
  특히 테스트 용이성을 해치지 않는 방향으로 설계합니다.
- TDD(Red-Green-Refactor)를 따릅니다.
- 구현 전에 계획을 구체화합니다.
- 계획을 구체화하는 과정에서 모호한 부분이 있으면, 작은 가설 코드나 실험으로 먼저 불확실성을 줄입니다.
- 모호한 내용이 남아 있으면 사용자에게 확인한 뒤 진행합니다.

## 테스트 및 Python 지침
- 테스트 커버리지는 유지보수 품질을 높이기 위한 수단으로 관리합니다. 커버리지를 높이기 위해 구현을 왜곡하거나 테스트를 해킹하지 않습니다.
- Python 코드를 수정하는 작업에서는 변경 범위에 대해 `uv run ruff check --fix <관련 경로>`와 `uv run pytest <관련 경로>`로 검증합니다.
- 검증은 항상 변경 범위에 맞는 최소 경로부터 시작하고, 영향 범위가 넓거나 커밋 전에는 필요한 범위까지 확장합니다.
- 테스트 코드를 작성할 때는 `pytest.MonkeyPatch`를 사용하지 않습니다.
- 테스트 가능성을 높이기 위해 의존성 주입을 우선적으로 사용합니다.
